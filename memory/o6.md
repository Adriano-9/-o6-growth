## 2026-07-07 · O6 Intelligence Diagnosis Engine v1 (app/_lib/diagnosis-engine/)

### Contexto
Explícito: O6 não está construindo SaaS pré-venda — é motor interno para acelerar entrega de consultoria real. O scoring existente (`app/offer-book/_lib/scores.ts`) é hardcoded ao schema fixo do Offer Book (6 scores presos a Cliente/ICP/Psicografia/Oferta/Diagnóstico) — não é reutilizável por vertical. Pedido explícito: **nunca duplicar o engine por indústria** — indústrias são só camada de configuração (pergunta/peso/recomendação/terminologia) sobre um núcleo único organizado por **capacidades de negócio**, não por setor.

### Decisão
`app/_lib/diagnosis-engine/`: `types.ts` (Capability/Question/Answer/VerticalConfig/DiagnosisResult), `capabilities.ts` (8 capacidades universais: aquisição, conversão, entrega, retenção, oferta&precificação, operação&eficiência, capacidade&time, saúde financeira — cada uma com 3-4 perguntas), `engine.ts` (única lógica: `scoreCapability`/`computeDiagnosis`, zero condicional por indústria), `helpers.ts` (normalizadores 0-100 extraídos/generalizados de `scores.ts`: clamp/filled/fillRatio reaproveitados, bucket de duração generalizado de `velocidadeScore`), `verticals/*.ts` (5 configs — generic, solo-service, local-retail, program-membership, unclassified — **só dados**, nenhuma lógica).

### Achado / bug corrigido durante implementação
Perguntas booleanas onde `true` é resposta ruim (ex.: "depende de 1 pessoa?", "sazonalidade forte?") pontuariam invertido com `scoreBoolean` ingênuo. Adicionado `invertBoolean?: boolean` ao `Question` type — confirmado corrigido via smoke test real (`capacidade.dependencia_unica=true` → score baixo, correto).

### Escopo v1 (decisão do usuário via AskUserQuestion)
Só a lib core — sem rota `app/api/**`, sem migration/tabela Supabase, sem UI. Os 4 casos reais citados (Patricia Abreu, Jhun, Empório, Xseries) não têm dado de negócio disponível nesta sessão — verticais criadas como **templates vazios/genéricos** (`TODO: recomendação real pendente do caso` explícito em todo texto de recomendação), sem inventar fatos sobre essas empresas. `program-membership.ts` (hipótese para Xseries) tem aviso explícito de que o modelo de negócio não foi confirmado.

### Verificação real
`npx tsx app/_lib/diagnosis-engine/__smoke__/run-example.ts` rodado de verdade — output JSON plausível, `overallScore: 43`, tier `low` em `capacidade_time` (29, correto — dependência de 1 pessoa). `npx tsc --noEmit -p .` limpo, zero erros.

### Não integrado (explícito, fora de escopo desta v1)
`app/offer-book/_lib/scores.ts` não foi tocado nem migrado — extração de helpers foi aditiva.

### Path tocado
`app/_lib/diagnosis-engine/{types,capabilities,engine,helpers}.ts`, `app/_lib/diagnosis-engine/verticals/{index,generic,solo-service,local-retail,program-membership,unclassified}.ts`, `app/_lib/diagnosis-engine/__smoke__/run-example.ts` (12 arquivos novos)

## 2026-07-06 · Patch de resiliência do Hermes bot (polling NetworkError)

### Contexto
Usuário reportou `telegram.error.NetworkError: httpx.ReadError` durante `get_updates` no bot Hermes (VPS 147.182.135.206) — long-polling caindo intermitentemente e derrubando o processo.

### Decisão
Timeouts HTTPX tolerantes no `Application.builder()` (30-40s vs. default ~5-10s), error handler global (`app.add_error_handler`) que loga `NetworkError`/`TimedOut` como `WARNING` em vez de deixar a exceção escapar, logging configurado (httpx/httpcore rebaixados para WARNING), e `StartLimitIntervalSec=0` na unit systemd para não esgotar o limite de restart do systemd sob rajada de falhas. Aplicado em dois lugares: `scripts/vps-patch-hermes-bot-resilience.sh` (patch idempotente com backup, para a instância já rodando) e `scripts/vps-bootstrap-hermes-bot.sh` (atualizado para que instalações futuras já venham resilientes — evita divergência entre bootstrap e patch).

### Trade-off
Não pinamos `python-telegram-bot` numa versão exata — só piso `>=20.0` (necessário para os métodos `get_updates_*_timeout()` do builder existirem). Não foi possível testar em produção nesta sessão (sem acesso SSH); script segue o padrão já validado (preparar → usuário cola no console → usuário confirma output real) antes de considerar aplicado.

### Não tocado
`/opt/o6-intelligence/**` (Intelligence Engine), `TELEGRAM_TOKEN`, `TELEGRAM_CHAT_ID` — conforme constraint explícito do pedido.

### Path tocado
`scripts/vps-patch-hermes-bot-resilience.sh` (novo), `scripts/vps-bootstrap-hermes-bot.sh` (atualizado), `docs/bootstrap/06_BOOTSTRAP_HERMES.md` (seção Troubleshooting nova)

## 2026-07-06 · Camada de integração central (app/_lib/integrations/)

### Contexto
Pedido explícito: transformar Hermes em orquestrador central, com conectores para Claude Code, Codex, GitHub, Supabase, Telegram, MCP Higgsfield, MCP 21.dev e Dashboard OS — cada um com interface única, config centralizada, autenticação, tratamento de erro, logs e documentação. Constraint explícito: "Não implemente agentes de negócio ainda."

### Decisão
Interface única `Connector` (`checkHealth()`, `isConfigured()`, `getRequiredConfig()`) em `app/_lib/integrations/types.ts`. Registry central (`registry.ts`) expõe `getAllConnectors()`/`getAllConnectorsHealth()` — ponto único que Hermes e o Dashboard OS consultam, sem precisar importar 8 módulos separados. 3 conectores reais (Supabase, Telegram, GitHub) fazem chamada de rede de verdade no `checkHealth()`. "Dashboard OS" não virou conector — é o consumidor do registry, não uma integração de si mesma.

### Achado durante a auditoria
4 dos 8 itens pedidos não têm API server-side chamável a partir do runtime Next.js: Claude Code (é o próprio agente rodando a sessão), Codex (nunca instalado neste repo), Higgsfield e 21st.dev (só existem como OAuth connector do claude.ai, sem API key própria — ver `docs/bootstrap/17_BOOTSTRAP_MCP.md`). Em vez de omitir ou fingir, criamos status `not_available` (diferente de `not_configured`) especificamente para essa distinção.

### Trade-off
Logs em memória (buffer local) seriam quase inúteis em produção — funções serverless da Vercel não compartilham memória entre invocações. Corrigido antes de finalizar: `logger.ts` persiste em `connector_logs` (migration `013_connector_logs.sql`), com fallback fail-soft ao buffer em memória só quando o Supabase está indisponível. Migration ainda não confirmada aplicada em produção — Supabase MCP falhou 3x com timeout de conexão nesta sessão; código já é fail-soft para funcionar mesmo sem a tabela.

### Verificado
`GET /api/integrations/status` testado via curl: `{ total: 7, connected: 1 (telegram), degraded: 1 (supabase, fetch failed), notConfigured: 1 (github, sem GITHUB_TOKEN), notAvailable: 4 }`. Seção "Integrações" nova em `/os` (`app/os/page.tsx`) renderizada e confirmada via screenshot com os 7 cards mostrando status real.

### Path tocado
`app/_lib/integrations/**` (novo), `app/api/integrations/status/route.ts` (novo), `app/os/page.tsx` (seção nova), `supabase/migrations/013_connector_logs.sql` (novo, aplicação pendente), `docs/bootstrap/17_BOOTSTRAP_MCP.md` (seção nova)

## 2026-06-03 · Seed inicial do OS

### Stack travada
- **Next.js 16** (App Router + Turbopack) — breaking vs. training data. Todo agente novo lê `node_modules/next/dist/docs/` antes de tocar routing/data.
- **React 19** · **TypeScript 5 strict** · **Tailwind 4** (`@theme` block em `app/globals.css`)
- **Supabase JS** via singleton lazy em `app/offer-book/_lib/supabase.ts` — única fonte de verdade pra dados de cliente.
- Deps extras: `framer-motion`, `lucide-react`. Nada mais.

### Supabase
- Projeto **O6 growth** · id `wphrwidjokimfjfvyaym` · region `us-west-2` · `ACTIVE_HEALTHY`.
- Env em `.env.local`: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` (publishable key `sb_publishable_…`).
- RLS habilitada em todas as tabelas, **policy `anon all`** intencional enquanto não houver auth UI. Apertar quando entrar login.
- Migrations versionadas `NNN_descricao_snake` via `mcp__supabase__apply_migration`. Nunca DDL via `execute_sql`.

### Verticais prioritários
Clínicas · Estética · Odontologia · Fisioterapia · Advogados. Default mental: dono de clínica de estética com ticket médio R$ 2.500. Toda copy/placeholder reflete esse universo.

### Design system (tokens em `app/globals.css`)
- `brand-orange #FF5722`, `brand-cyan #00E5FF`, `brand-graphite #121212`, `brand-offwhite #F8F9FA`.
- Superfícies dark do OS: `bg-zinc-950` (canvas), `bg-zinc-900/40` (cards glass), `border-white/10`, `bg-white/[0.04]` (hover).
- Tier colors: `emerald-400` (≥70), `amber-300` (40-69), `red-400/500` (<40).
- Tipografia: Geist Sans Black 900 uppercase `tracking-tight` pra titles; labels micro `text-[10px] uppercase tracking-[0.18em]`.
- 6 pilares: Dark · SaaS Premium · Bento Grid · Glassmorphism · Mobile First · Conversão.

### Padrões compartilhados
- Path alias `@/*` → `./` (configurado em `tsconfig.json`).
- Diretórios privados com prefixo `_` (`_lib/`, `_components/`) — Next.js ignora pra routing.
- Mappers camelCase ↔ snake_case isolados em `_lib/store.tsx` por módulo. Quando uma tabela for compartilhada, mover pra `_lib/db.ts` global.

### Não-objetivos explícitos
- **Sem auth UI** (RLS anon all é decisão consciente).
- **Sem IA na lógica** — scores e plano de ação são 100% determinísticos. Política revisada quando ROI de IA estiver mapeado.
- **Sem lib de PDF** (jsPDF, react-pdf, puppeteer). `/print` usa `window.print()` + `@media print` CSS.
- **Sem state manager global** (Redux/Zustand/Jotai). Cada módulo controla state local; cross-module só via Supabase.
- **Sem CSS-in-JS**. Tailwind utility + `globals.css` only.

### Sub-rules culturais
- Bias para ação. Auto Mode é o default; o usuário redireciona quando precisa.
- `AskUserQuestion` só pra decisão que só o usuário toma. Nunca pra "posso prosseguir?".
- Verificação obrigatória pós-mudança: screenshot via Preview MCP (UI) + `execute_sql` (data).
- TaskCreate quando trabalho cruza ≥ 3 passos distintos.

### Trade-offs aceitos
- RLS permissivo = qualquer um com a anon key lê/escreve tudo. OK porque a URL/key não está pública e o uso é interno por enquanto.
- Mappers escritos à mão em vez de gerar tipos via `generate_typescript_types` — economia agora, dívida pequena. Gerar quando ≥ 6 tabelas.
- `window.print()` em vez de PDF lib = depende do browser do usuário renderizar bem. OK porque o cliente recebe o PDF final pelo navegador dele.
